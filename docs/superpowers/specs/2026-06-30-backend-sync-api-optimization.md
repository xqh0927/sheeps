# 后端 Sync 接口批处理优化设计方案

## 需求背景
当前客户端同步用户数据时，调用的接口 `POST /api/user/sync` 响应时间长达 7000ms+。
经排查，该接口在处理同步请求时，存在以下性能瓶颈：
- **大量顺序数据库查询（N+1 问题）**：代码中依次、单条执行多次 `env.DB.prepare(...).run()`，包括获取历史数据（3次查询）、写备份日志（1次）、删历史备份（1次）、更新积分（1次）、循环插入关卡（N次）、循环更新道具（M次）以及获取最新状态（3次）。
- 每次查询都与 Cloudflare D1 产生一次网络往返（Network Roundtrip），导致累积延迟极高。

---

## 优化设计
Cloudflare D1 提供了 `env.DB.batch([...])` 方法，可以在单次数据库事务和单次网络交互中，并行执行多个 SQL 语句。我们计划将原有的顺序 DB 交互重构为 **3 个批处理阶段**：

1. **阶段一：并行拉取旧数据（Old Data Fetch）**
   - 合并查询旧的 `users` 积分、`level_unlock` 解锁关卡和 `user_items` 背包道具。
   - 网络往返次数：**3 次 -> 1 次**。

2. **阶段二：事务性批量写入（Mutations Write Batch）**
   - 合并备份日志写入、过期备份删除、积分更新、多条关卡插入及多条道具更新。
   - 所有写入语句（Mutation Statements）放入一个数组，通过单个 `env.DB.batch()` 提交。
   - 网络往返次数：**3 + N + M 次 -> 1 次**。

3. **阶段三：并行拉取最新数据（Updated Data Fetch）**
   - 合并查询更新后的 `users`、`level_unlock` 和 `user_items` 状态，返回给客户端。
   - 网络往返次数：**3 次 -> 1 次**。

重构后，无论同步的关卡数和道具数有多少，接口的数据库网络交互次数**固定为 3 次**，可大幅减少网络等待开销，预计能将响应时间降低至 150-300ms 左右。
